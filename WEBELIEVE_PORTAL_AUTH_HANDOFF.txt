Authorization model
- JWT auth is implemented with `@fastify/jwt`.
- Route guard plugin:
  - `/home/keycloakadmin/shared/webelieveacademics.com/frontend/src/plugins/auth.ts`
- Decorators:
  - `fastify.authenticate`
  - `fastify.authorizeAdmin`
- `authorizeAdmin` verifies the JWT and requires `request.user.role === 'admin'`

Current account creation flow
1. Parent registration
- Route: `POST /api/v1/auth/register-parent`
- Source: `src/routes/auth.ts`
- Validates:
  - `email`
  - `password` min 8
  - `firstName`
  - optional `lastName`, `displayName`, `phone`
  - `preferredContact` in `email | phone | text`
- Database writes:
  1. `users`
  2. `parents`
  3. `auth_local_credentials`
- Token:
  - JWT is issued immediately after registration
- Result:
  - parent is created and logged in immediately

2. Login
- Route: `POST /api/v1/auth/login`
- Source: `src/routes/auth.ts`
- Reads:
  - `users`
  - `auth_local_credentials`
- Checks:
  - user exists
  - `users.is_active = true`
  - password hash matches
- Writes:
  - updates `users.last_login_at`
- Token:
  - JWT is returned on success

3. Current student creation flow
- Route: `POST /api/v1/students`
- Source: `src/routes/students.ts`
- Requires JWT auth
- Validates:
  - `email`
  - `firstName`
  - optional `lastName`, `displayName`, `gradeLevel`, `dateOfBirth`, `parentId`, `schoolName`, `notes`
- Database writes:
  1. `users` with role `student`
  2. `students`
- Parent linkage:
  - `students.parent_id` defaults to `request.user.sub` unless `parentId` is supplied

Portal data reads
1. Parent-linked student list
- Route: `GET /api/v1/parents/me/students`
- Source: `src/routes/students.ts`
- Allowed roles:
  - `parent`
  - `admin`
- Reads:
  - `students`
  - `users`
- Important limitation:
  - the query uses `where s.parent_id = request.user.sub`
  - this works correctly for parents
  - for admins, it will only show students whose `parent_id` equals the admin user id, which is usually none
  - so the current admin role is allowed by policy but not truly useful for this endpoint

2. Student dashboard
- Route: `GET /api/v1/dashboard/student/:studentId`
- Source: `src/routes/dashboard.ts`
- Requires JWT auth
- Reads:
  - `users`
  - `students`
  - `student_enrollments`
  - `courses`
  - `grades`
  - `tests`
  - `student_learning_preferences`
  - `teaching_methods`
- Important gap:
  - there is no ownership check ensuring a parent can only load their own student
  - any authenticated user with a student UUID can currently query that student dashboard
  - this should be fixed

Live database snapshot at time of handoff
- Query results from the live local Postgres instance:
- Role counts:
  - `admin`: 1
  - `parent`: 3
  - `student`: 3
- Table counts:
  - `consultation_requests`: 3
  - `auth_local_credentials`: 4
  - `students`: 3
  - `parents`: 3

Interpretation of those counts
- There is 1 admin account with local credentials
- There are 3 parents with local credentials
- Students currently do not have their own local credentials
- Student rows are created as user records plus student profile rows, but not login credentials
- That is why `auth_local_credentials` count is 4 while `users` total is 7

Known functional gaps another AI should address next
1. Student/teacher/admin account lifecycle
- Add admin CRUD routes to create:
  - admins
  - teachers
  - students with optional login credentials
  - parents manually
- Add ability to issue or reset passwords from admin

2. Authorization hardening
- Restrict `GET /dashboard/student/:studentId` so:
  - parent can only access their own linked students
  - admin can access all
  - teacher rules are explicit
- Decide whether `GET /parents/me/students` should really allow admins, and if yes, add an admin-specific variant

3. Refresh tokens and session model
- `api_refresh_tokens` table exists but the flow is not implemented
- Current portal uses only one JWT stored in localStorage
- Consider:
  - short-lived access token
  - refresh token rotation
  - logout invalidation

4. Password reset / email verification
- Missing today
- Needed for a real production auth system

5. Teacher portal
- No teacher-facing auth/UI flow exists yet

6. Admin UX
- Admin dashboard exists in `portal.html`
- It is functional but still basic
- Next major additions:
  - user creation forms
  - password reset actions
  - student-parent reassignment
  - teacher assignment to courses
  - richer consultation CRM workflow

7. Portal code structure
- Current portal is still a static HTML + JS page:
  - `portal.html`
  - `assets/app.js`
- It works, but longer term it should likely be migrated into the Next frontend so auth/admin UX is not split between:
  - `web/` marketing frontend
  - separate static portal
